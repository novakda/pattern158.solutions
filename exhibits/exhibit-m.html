EXHIBIT M: SCORM DEBUGGER — TASBOT FOR ELEARNING

Automated Testing Architecture — QA Cycles from Hours to Minutes

Classification: Field Report
Client: GP Strategies (Internal Tooling)
Location: Internal Development
Period: ~2019–2020
Investigator: Dan Novak, Computer Programmer, GP Strategies
Field Report · Exhibit M · GP Strategies (Internal Tooling)
1. Background

During QA at GP Strategies, testing gated eLearning courses — both completion-gated and assessment-gated — required manually clicking through every section to reach the part needing testing. Hours per QA cycle, per bug, per regression check. With hundreds of courses in production across clients, the QA burden was enormous but invisible — counted as “just part of the job” in a direct-labor billing model.

The inspiration came from speedrunning and TASBot culture: emulator save states as the model. Dan recognized that course state management was the same problem speedrunners had solved decades ago. Instead of replaying from the beginning every time, save the state and restore it instantly.

2. Personnel Involved

Dan Novak created and developed the SCORM Debugger, handling architecture design, Vue.js implementation, SCORM API emulation, and tool deployment.

Max Glick, Engineering Manager, provided team advocacy and engineering group leadership support for the project.

3. Sequence of Events
QA Pain Point
Manual Playthrough Bottleneck

Testing gated courses required full manual playthrough to reach test targets. Hours per cycle, repeated for every bug and every regression check across hundreds of courses in production.

The Parallel
Emulator Save State Model

Dan recognizes the emulator save state model from speedrunning and TASBot culture. Course state management is the same problem — instead of replaying from the beginning every time, save the state and restore it instantly.

Architecture
Vue.js Wrapper with SCORM Emulation

Vue.js wrapper with open-source SCORM API emulation library providing a fake SCORM runtime environment. Courses run inside the wrapper as if they were in a real LMS — but with full state control.

Core Features
Save, Restore, Edit, Inspect

Save course state at any point. Restore previous state for instant replay. Manually edit SCORM values (completion status, scores, suspend data). Jump directly to any section. Inspect real-time SCORM communication.

Adoption
Team Adoption and Impact

Some team members adopted the tool for their QA workflows. Testing cycles dropped from hours to minutes for gated content.

The Wall
Business Model Barrier

GP Strategies’ direct-labor, billable-hours-first business model prevented investing non-billable time in internal tooling, even though it would have paid for itself many times over in QA savings.

4. Findings
# Finding Description
F-01 QA Time Sink Testing gated courses consumed hours per cycle — manual clickthrough of every section to reach test target, repeated for every bug and regression check.
F-02 State Management Parallel Course state management is isomorphic to emulator save states — the same problem speedrunners solved decades ago with different tools.
F-03 Tool Effectiveness Save/restore reduced testing cycles from hours to minutes. SCORM value editing eliminated the need for full course playthrough to test specific scenarios.
F-04 Adoption Barrier GP’s direct-labor billing model treated internal tooling as unbillable overhead — no budget category existed for “tools that make everyone faster.”
F-05 Rustici Gap SCORM Cloud and SCORM Engine are excellent products, but reproducible debug states and session state inspection are not first-class features. This tool addressed exactly that gap from the integrator side.
5. Probable Cause

The SCORM Debugger was never fully polished or formally adopted — not because the tool didn’t work, but because GP Strategies’ organizational structure had no mechanism for investing in internal tooling. The direct-labor, billable-hours-first business model prevented allocating non-billable time to polish and deploy productivity tools, even when the ROI was obvious. Dan spent more time developing it than using it because there was never enough uninterrupted time to both build and deploy it properly.

This is not a failure story — it’s a culture-fit diagnostic. Organizations that invest in developer productivity and internal tooling are the right environment for this kind of work. The SCORM Debugger functions as a signal: when Dan tells this story in interviews, the listener’s reaction reveals whether their organization values engineering investment or treats tooling as overhead.

6. Outcome

What worked: Testing cycles dropped from hours to minutes for gated content. Some team members adopted the tool and used it productively.

What didn’t: The tool was never fully finished or formally adopted across the team. The business model prevented the investment that would have made it a standard part of the QA workflow.

The Rustici connection: SCORM Cloud and SCORM Engine provide excellent runtime environments, but reproducible debug states — the ability to save, restore, and edit course state for targeted testing — are not first-class features in any commercial SCORM product. This tool was purpose-built to fill that gap from the integrator’s perspective.

7. Technologies
Category Technologies & Tools
Frontend Framework Vue.js (component architecture, reactive state management)
SCORM Integration SCORM 1.2 API emulation, SCORM 2004 API emulation (open-source library)
Core Languages JavaScript (ES6+), HTML5, CSS3
State Management Course state serialization/deserialization, localStorage persistence, SCORM suspend_data manipulation
Debugging Real-time SCORM API call inspection, value editing, completion status manipulation
Inspiration Tool-Assisted Speedrunning (TASBot), emulator save states, frame-perfect optimization methodology

Key Technical Insights

  • Save State Architecture: Course state serialized to JSON snapshots — restore any previous state instantly without replaying from beginning
  • Fake SCORM Runtime: Open-source SCORM API emulation provided complete runtime environment without requiring LMS — courses couldn’t tell the difference
  • The TASBot Parallel: Just as tool-assisted speedrunners use save states to achieve frame-perfect optimization, SCORM Debugger used state snapshots to achieve targeted, reproducible QA — turning hours of manual testing into minutes of precision work
  • Integrator-Side Gap: Commercial SCORM tools (SCORM Cloud, SCORM Engine) excel at runtime and conformance testing, but reproducible debug states and session-level state inspection remain underserved — this tool addressed that specific gap

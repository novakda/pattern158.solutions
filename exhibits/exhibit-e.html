EXHIBIT E: CSBB DISPATCH

Cross-Domain Content Delivery — Built 5 Years Before Content Controller

Client: GP Strategies (Multiple Energy Utility Clients)
Period: 2011–2016+ (10+ years in production)
Team: Dan Novak (Architect/Developer), Zach Taylor (GPiLEARN Developer), Ron Cyran (Program Manager), Tom Pizer (Director of Learning Technologies)
Field Report · Exhibit E · GP Strategies · 2011–2016+

“Built in 2011 — five years before Rustici released Content Controller to solve the same problem.”

1. Background

Version control nightmare: every course update required emailing zip files to 7+ energy utility clients for manual re-upload to their respective LMS platforms. Different versions of the same content running on different client systems, with no centralized control over what was deployed where. A single course update could take days to propagate across all client deployments.

The problem wasn’t just logistics — it was architectural. Each client’s LMS enforced same-origin security policies, making direct cross-domain content hosting impossible with standard approaches. The content needed to live on GP Strategies’ servers for centralized control, but execute inside each client’s LMS domain for SCORM communication. No commercial product existed to solve this problem in 2011.

2. Personnel Involved

Dan Novak served as architect and developer, designing the system architecture, cross-domain communication layer, SCORM API proxy, and package generation tooling.

Zach Taylor was the primary development collaborator on the GPiLEARN platform, handling platform engineering. Ron Cyran served as Program Manager, providing GPiLEARN program management and client coordination. Tom Pizer, Director of Learning Technologies, provided technical direction and managed the hosting infrastructure.

3. Sequence of Events
The Problem
Version Control Nightmare

Course updates require emailing zip files to 7+ energy clients for manual re-upload. Version chaos across deployments.

Architecture Design
Centralized Hosting with Stub Packages

Dan designs centralized content hosting on GP servers with lightweight “stub” packages deployed to client LMS platforms. Stubs are kilobytes, not megabytes.

Cross-Domain Bridge
EasyXDM Communication Layer

JavaScript/CORS communication layer using EasyXDM to bridge the cross-domain gap between GP’s content servers and client LMS domains.

Protocol Translation
SCORM API Proxy

SCORM API proxy with transparent AICC/SCORM protocol translation — the same content works regardless of whether the client LMS speaks SCORM or AICC.

CSBB Generator
Automated Package Generation

Package generation tooling (Groovy, Access, Windows Forms) automating the creation and deployment of stub packages. Global deployments without touching individual client systems.

Scale
Multi-Client Deployment

Deployed across 7+ energy utility clients: FPL, Puget Sound Energy, NRG, Colorado Springs Utilities, Calpine, Exelon, BP. 100+ packages in production.

Defense Adaptation
Cross-Industry Reuse

Architecture later adapted for General Dynamics Electric Boat defense sector deployment, proving the pattern was reusable across industries.

Validation
Market Catches Up

System remained in production for 10+ years. In 2016, Rustici Software released Content Controller to solve the same centralized dispatch problem commercially.

4. Findings
# Finding Description
F-01 Cross-Domain Architecture EasyXDM-based communication layer bridged same-origin policy constraints between GP content servers and client LMS domains — transparent to both content and LMS.
F-02 Protocol Agnostic SCORM API proxy provided transparent AICC/SCORM protocol translation — identical content packages worked across LMS platforms regardless of protocol dialect.
F-03 Deployment Revolution Stub packages (kilobytes) replaced full content packages (megabytes) on client LMS. Course updates deployed centrally, propagated instantly to all clients.
F-04 Automation at Scale CSBB Generator tooling (Groovy, Access, Windows Forms) automated package creation for 100+ courses across 7+ clients — global deployments without touching individual systems.
F-05 Cross-Industry Reuse Architecture originally built for energy utilities was successfully adapted for defense sector (General Dynamics Electric Boat) — proving the pattern was reusable, not domain-specific.
5. Probable Cause

CSBB Dispatch solved a problem that had no commercial solution in 2011: centralized cross-domain content delivery with transparent protocol translation. The architecture — lightweight stubs on client LMS platforms communicating with centralized content servers via EasyXDM — eliminated the version control nightmare of distributing zip files to multiple clients and enabled instant, centralized updates across all deployments.

Five years later, in 2016, Rustici Software released Content Controller to address the same centralized dispatch problem. CSBB was built five years before the commercial solution existed. It solved a real architectural problem using the tools available, and remained in production for over a decade.

6. Outcome

The system served 7+ major energy utility clients (FPL, Puget Sound Energy, NRG, Colorado Springs Utilities, Calpine, Exelon, BP) with 100+ packages in production. It remained operational for 10+ years — outlasting most of the projects it was built to support.

The architecture proved reusable when adapted for General Dynamics Electric Boat’s defense sector deployment, demonstrating that the cross-domain dispatch pattern wasn’t a one-off solution but a generalizable approach to multi-tenant content delivery.

Dan encountered the same architectural pattern a decade later on the BP Learning Platform (2024) — a federated integration layer (React/GraphQL) stitching together Rustici Content Controller, PeopleFluent LMS, Watershed LRS, and Amazon Cognito. He didn’t build that platform, but he recognized the architecture immediately because he’d invented his own version of it in 2011.

Key Technical Insights

  • Stub Architecture: Lightweight packages (kilobytes) deployed to client LMS contain only launch and communication code — actual content served from centralized GP servers, enabling instant updates without client-side redeployment
  • Protocol Translation Layer: SCORM API proxy transparently converts between SCORM and AICC protocols — content authors write once, the dispatch layer handles LMS dialect differences
  • Predating the Market: Built in 2011, five years before Rustici Content Controller (2016). The same centralized dispatch problem, solved independently with the tools available at the time
  • Cross-Industry Pattern: Architecture proved reusable from energy utilities to defense sector — the cross-domain dispatch pattern is domain-agnostic

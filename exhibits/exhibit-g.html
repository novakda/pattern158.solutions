EXHIBIT G: SUNTRUST AWARE PLATFORM

Custom Integration Architecture for Multi-Year Security Training Program

Classification: Custom Integration / SCORM-to-API Wrapper
Client: SunTrust Bank (via GP Strategies)
Location: United States
Period: 2015–2018
Investigator: Dan Novak, Computer Programmer, GP Strategies
Field Report · Exhibit G · SunTrust Bank (via GP Strategies)
1. Background

SunTrust Bank (now Truist Financial, formed from the SunTrust/BB&T merger in 2019) required a specialized compliance and onboarding training platform for external learners—individuals who were not SunTrust employees but required bank-certified training to conduct business with the institution.

The AWARE platform could not use SunTrust's standard internal LMS infrastructure. External learners operated outside the corporate network and identity management systems, yet required the same rigorous compliance tracking and completion verification as internal staff. This created a unique architectural challenge: deliver SCORM-based courseware through a proprietary web application without a traditional Learning Management System.

The solution required building a custom integration layer that would translate between SCORM courseware (Lectora, Adobe Captivate) and SunTrust's proprietary web service API—replacing the entire LMS stack with a purpose-built wrapper that handled status tracking, score reporting, and bookmark/state persistence through direct API calls.

2. Personnel

Dan Novak served as Lead Technical Engineer for the integration architecture, coordinating with the SunTrust developer responsible for API and backend development.

On the GP Strategies side, the Project Manager/Account Lead handled SunTrust account management and engagement coordination. The Director of Learning Technologies provided technical leadership and oversight. Additional GP Strategies team members contributed to instructional design and quality assurance.

3. The Integration Challenge

Standard SCORM courseware expects to communicate with an LMS through the SCORM API adapter (JavaScript runtime). The LMS tracks learner progress, stores scores, manages bookmarks, and handles completion status. Removing the LMS from this equation meant rebuilding that entire communication layer.

Architectural Requirements

The custom SCORM-to-API wrapper needed to:

  • Intercept SCORM Runtime Calls: Capture LMSInitialize, LMSSetValue, LMSGetValue, LMSCommit, and LMSFinish calls from Lectora/Captivate courseware
  • Translate to REST API: Convert SCORM data model elements to SunTrust's proprietary web service endpoints
  • Persist State: Handle cross-session resume functionality using SunTrust's backend storage rather than LMS session management
  • Cross-Computer Resume: Enable learners to start training on one machine and resume on another—requiring server-side state management instead of local browser storage
  • Fallback Mechanism: Implement cookie-based fallback for scenarios where API connectivity was degraded
  • Browser Expansion: Support modern browsers (Chrome, Firefox) in addition to legacy IE8/IE10 requirements

Security and Trust Implications

The integration required such deep access to SunTrust's systems that Dan was issued a dedicated SunTrust laptop and employee credentials (Daniel.A.Novak@SunTrust.com)—extraordinary measures for an external contractor. This level of trust reflected both the sensitivity of the banking institution's systems and the confidence in Dan's ability to operate within those constraints responsibly.

4. Technical Approach

Dan scoped a 66-hour integration lifecycle covering the full development process. The approach built on GP Strategies' existing SCORM-to-AICC wrapper technology, adapted for SunTrust's proprietary REST API.

Five-Phase Integration Lifecycle

  1. API Development Consultation: Work directly with SunTrust's backend developer during API design phase to ensure the endpoints could handle SCORM data model requirements (status values, score normalization, suspend_data size limits, interaction tracking)
  2. API Definition Agreement: Formalize endpoint contracts, authentication mechanisms, error handling protocols, and data format specifications—establishing the interface contract between courseware wrapper and banking backend
  3. Direct API Testing (Fiddler): Use HTTP debugging tools to test raw API calls independent of courseware, validating authentication, data serialization, edge cases, and error responses before integration
  4. SCORM Call Monitoring: Instrument Lectora lessons to capture actual SCORM runtime behavior—identifying which data model elements were set, in what sequence, and with what values during real courseware execution
  5. Wrapper Adaptation: Modify GP's existing SCORM-to-AICC bridge to route calls through SunTrust's API instead of AICC HACP endpoints, implementing mapping logic, retry handling, and the cookie-based fallback mechanism

Key Architectural Decisions Dan Shaped

During the scoping and consultation phase, Dan raised critical questions that directly influenced the final architecture:

  • Cross-Computer Resume Requirements: Clarified whether learners needed the ability to start training on one device and continue on another—a requirement that mandated server-side state storage rather than client-side browser caching
  • Security Implications of Backend Modifications: Identified potential risks in exposing training data through web services and recommended authentication patterns appropriate for external learner access
  • Development and Testing Environment Needs: Specified the requirement for proper dev/test/prod environment separation to allow safe integration development without risking production training delivery

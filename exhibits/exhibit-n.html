EXHIBIT N: BP LEARNING PLATFORM

Federated System Integration — Pattern Recognition Across 13 Years

Client: BP (via Leo Learning / GP Strategies)
Period: 2024
Team: Dan Novak (Frontend Developer / Integration Engineer)
Field Report · Exhibit N · BP · 2024
1. Background

The BP Learning Platform was a React/GraphQL frontend acting as a federated facade over multiple backend systems: Rustici Content Controller for content playback, PeopleFluent LMS for learning management, Watershed LRS for xAPI analytics, and Amazon Cognito for authentication. Dan worked directly with Rustici Content Controller as an integrator and client-side developer.

Dan’s central challenge was a large-scale multi-tenant rebranding that required navigating the platform’s largely undocumented Material UI theming system. Initially scoped as a one-week task, it expanded into a several-month effort as theming complexity revealed deeply interrelated style dependencies across the entire component library. This was Dan’s first brownfield project with well-organized engineering practices (JIRA, defined acceptance criteria) from the acquired Leo Learning group.

The platform was built by the Leo Learning team before GP Strategies acquired them — Dan joined as a developer on an existing, well-architected system. But the architectural pattern was immediately familiar: another federated integration layer stitching multiple backend systems through a single frontend, echoing the CSBB Dispatch approach Dan had built a decade earlier. He recognized the pattern because he’d invented his own version of it in 2011.

2. Personnel Involved

Dan Novak served as Frontend Developer and Integration Engineer, handling Material UI theming, bug fixes, Cognito troubleshooting, Watershed LRS query analysis, and component development. This work was performed as part of the Leo Learning team after their acquisition by GP Strategies.

5. Probable Cause

The BP Learning Platform exemplifies federated system integration — the architectural challenge of making multiple independent systems behave as a coherent whole. The platform’s React/GraphQL frontend served as a unifying facade over Rustici Content Controller (content playback), PeopleFluent LMS (learning management), Watershed LRS (xAPI analytics), and Amazon Cognito (authentication).

The rebranding challenge exposed a truth about brownfield integration work: when systems are undocumented and deeply interrelated, every change becomes an archaeological expedition. Material UI’s theming cascades meant that modifying one component’s appearance could ripple through dozens of others in unpredictable ways. The only reliable approach was systematic exploration — the same forensic methodology Dan applied to SCORM protocol debugging and LMS architecture analysis throughout his career.

6. Outcome

Dan successfully navigated the theming challenge through systematic exploration of the undocumented Material UI system, delivered bug fixes across the React/GraphQL frontend, troubleshot Amazon Cognito authentication issues through log analysis, and reverse-engineered Watershed LRS queries for xAPI analytics.

The architectural through-line is the most significant observation: CSBB Dispatch (2011) and the BP Learning Platform (2024) represent the same architectural pattern thirteen years apart — a unifying integration layer that makes multiple backend systems behave as one. Dan built the 2011 version from scratch; he worked inside the 2024 version as a developer. In 2011, it was EasyXDM bridging SCORM content servers and client LMS platforms. In 2024, it was React/GraphQL bridging Rustici, PeopleFluent, Watershed, and Cognito. Different authors, same pattern. Dan recognized it because he’d invented his own version of it.

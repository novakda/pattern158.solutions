EXHIBIT O: CONTENTAIQ

The Integration Thread — Pattern Recognition Across Three Projects

Classification: Field Report
Client: GP Strategies (internal product)
Location: Cloud-based multitenant SaaS
Period: 2024–2025
Investigator: Dan Novak, Computer Programmer, GP Strategies
Field Report · Exhibit O · GP Strategies (internal product)
1. Background

Integration expertise as the organic path to AI product work: three GP Strategies projects (BP Learning Platform, AICPA Bridge Adapter, ContentAIQ) unified by a common thread of making multiple systems work together. The progression wasn’t deliberate career planning — these were assignments that came through the same employer — but the skill thread was real. Each project represented integration complexity at increasing levels of sophistication.

The BP Learning Platform was a React/GraphQL facade over four independent systems (Rustici Content Controller, PeopleFluent LMS, Watershed LRS, Amazon Cognito). Dan recognized this federated architecture because he’d built his own version as CSBB Dispatch a decade earlier — same pattern, different tooling.

The AICPA Bridge Adapter was an integration layer Dan co-maintained with another developer, supporting the original senior engineer: a webhook-driven sync pipeline moving curriculum and registration data from AirTable through Node.js to Bridge LMS. Dan became an L3 escalation point for complex sync failures, co-owning a multi-system integration he didn’t build.

ContentAIQ is GP Strategies’ AI-powered learning content creation platform — a cloud-based, multitenant SaaS product that generates course outlines, scripts, assessments, metadata, and gamification elements using AI agents with built-in instructional design expertise. Dan was one of several frontend developers in the engineering group building the UX: implementing new React components and pages from Adobe XD designs, modifying UX flows across the multitenant architecture. An early-stage product in pilot program, rapidly iterating toward production.

The thread connecting all three projects is integration expertise — making multiple systems work together, whether that’s SCORM content servers, LMS platforms, AirTable data, or AI agents. This thread naturally led to AI product work because AI products are fundamentally integration challenges: the frontend surfaces capabilities generated by backend AI pipelines, requiring the same cross-system coordination and debugging skills honed on traditional integrations.

2. BP Learning Platform — Federated Facade

The BP Learning Platform represents the first point in the arc: a federated React/GraphQL frontend integrating Rustici Content Controller, PeopleFluent LMS, Watershed LRS, and Amazon Cognito into a coherent whole. (Full details in Exhibit N.)

Dan’s central challenge was a multi-tenant rebranding through Material UI’s largely undocumented theming system — initially scoped as one week, it expanded to several months as deeply interrelated style dependencies revealed themselves. But the architectural signal is more significant: Dan recognized this federated facade pattern because he’d invented his own version as CSBB Dispatch in 2011. Different authors, same pattern. In 2011, it was EasyXDM bridging SCORM content servers; in 2024, it was React/GraphQL bridging enterprise backend services.

Key signal: Pattern recognition across thirteen years — the same architectural approach to making multiple independent systems behave as one.

3. AICPA Bridge Adapter — Webhook Sync Pipeline

The AICPA Bridge Adapter represents the second point in the arc: an integration layer Dan inherited rather than built. A Node.js plugin syncing curriculum and registration data between AirTable (AICPA’s source of truth) and Bridge LMS (course delivery platform), using webhooks to trigger sync operations and PostgreSQL to track state across subaccounts.

Data flow: AirTable fires webhook → Node.js receives webhook and pulls full record via AirTable API → PostgreSQL tracks sync state → Node.js builds and executes Bridge REST API calls to create/update curricula, courses, enrollments. The PostgreSQL layer compensates for the Bridge API’s inability to track what’s been synced where per subaccount.

Dan was one of two developers supporting the original senior engineer who built the integration — inherited code stewardship as a team. Working under senior architect guidance, they fixed bugs in the webhook-driven sync logic, improved error handling, and troubleshot failed sync issues. Dan served as an L3 escalation point when complex failures escalated through support tiers, and worked on the React/Material UI admin dashboard component.

Key signal: Co-ownership of a complex multi-system integration built by others — the ability to step into existing codebases, understand architectures quickly, and maintain production systems reliably as part of a small support team.

4. ContentAIQ — AI Product Frontend

ContentAIQ is the destination: GP Strategies’ AI-powered learning content creation platform, a cloud-based multitenant SaaS product built on Microsoft Azure. The product uses AI agents with built-in instructional design expertise to generate course outlines, scripts, assessments, metadata, gamification elements, and more. (See official product page.)

Dan was one of several frontend developers in the engineering group working on the UX that surfaces these AI capabilities: implementing new React components and pages from Adobe XD designs, modifying existing UX flows across the multitenant architecture. Framed honestly, Dan contributed to the interface for an AI product — he didn’t build the AI pipeline itself, and he wasn’t the sole frontend developer, but he was part of the team building the components users interact with to access AI-generated content.

The technical architecture is a Turborepo monorepo built with Next.js, React, TypeScript, and Tailwind CSS, deployed on Azure. The multitenant architecture was designed for per-client theming, but during Dan’s tenure the system was still on the base UX — work focused on getting the shared foundation right before client-specific branding.

Design translation challenge: UX was designed by a graphic artist using Adobe XD (not a UX/interaction designer), which created translation challenges from static layouts to interactive components. The designs often didn’t account for interactive states, responsive behavior, or component reusability — the engineering team had to translate visual mockups into pragmatic React implementations that worked across devices and usage patterns.

Product stage: Early-stage product in pilot program with select clients. UX in heavy flux, designs changing frequently, rapid iteration toward production. This environment required frontend agility — building features quickly while maintaining architectural flexibility for future pivots.

Key signals: AI product work (contributing to the interface for LLM-powered content generation), full product lifecycle experience (pilot-stage rapid iteration), enterprise-scale React (Next.js monorepo, multitenant SaaS), and design-to-code pragmatism (translating imperfect handoffs into working software).

6. Probable Cause

The thread connecting BP Learning Platform, AICPA Bridge Adapter, and ContentAIQ is integration expertise — the ability to make multiple systems work together as a coherent whole. Whether that’s SCORM content servers and LMS platforms, AirTable data and Bridge LMS, or AI agents and frontend interfaces, the underlying challenge is the same: coordinating across boundaries, debugging failures that span systems, and building layers that unify disparate components.

This thread naturally led to AI product work because AI products are fundamentally integration challenges. The AI pipeline generates content; the frontend surfaces it. The AI agent produces outputs; the UI translates them into user-facing features. Contributing to the interface for an AI product as part of an engineering team requires the same cross-system coordination, debugging across boundaries, and architectural thinking that Dan applied to federated LMS facades and webhook sync pipelines.

The progression wasn’t deliberate career planning — these were assignments from the same employer, not a pre-planned trajectory. But the skill thread was real, and pattern recognition mattered: Dan recognized the BP Learning Platform’s federated facade because he’d built his own version a decade earlier. The same patterns, expressed in increasingly sophisticated contexts.
